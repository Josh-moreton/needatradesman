# Infinite Redirect Loop - Root Cause Analysis

## TL;DR
**The session token in Clerk Dashboard is NOT configured to include `publicMetadata`**, so the middleware sees `sessionClaims.publicMetadata === undefined` and thinks the user isn't onboarded, causing infinite redirects.

## What's Happening

### Current Flow (Broken)
1. User signs up → API sets `publicMetadata.onboardingComplete = true` ✅
2. User calls `user?.reload()` → Gets fresh user object ✅  
3. User navigates with `router.push('/dashboard')` ✅
4. **Middleware checks `sessionClaims.publicMetadata`** ❌ **Returns `undefined`**
5. Middleware thinks user isn't onboarded → Redirects to `/onboarding`
6. **LOOP**: Onboarding page → selects role → API succeeds → reload → navigate → middleware fails → back to onboarding

### Why Main Branch Works
The main branch has this fallback code that we removed:

```typescript
// If session claims don't have metadata, fetch fresh data as fallback
if (userId && !('publicMetadata' in (sessionClaims || {}))) {
    const { clerkClient } = await import('@clerk/nextjs/server')
    const client = await clerkClient()
    const freshUser = await client.users.getUser(userId)
    onboarded = !!freshUser.publicMetadata?.onboardingComplete
}
```

This fetches fresh user data **on every middleware request** when `publicMetadata` is missing from the JWT. It's slow and defeats the purpose of JWTs, but it **works around the missing session token configuration**.

## The Real Problem

**Clerk's session token doesn't include `publicMetadata` by default.** You must configure it in the Dashboard.

### What We Removed (and Why It Seemed to Work)

We removed **110 lines** of:
- Fetching fresh user data on every request
- Complex error handling
- Retry logic
- Fallback mechanisms

This was all **compensating for a missing Clerk Dashboard configuration**.

### What Clerk's Official Docs Say

From https://clerk.com/docs/guides/development/add-onboarding-flow:

> **Add custom claims to your session token**
> 
> Session tokens are JWTs that are generated by Clerk on behalf of your instance. With Clerk, when a session token exists for a user, it indicates that the user is authenticated.
> 
> For this guide, you will use an `onboardingComplete` property in the user's public metadata to track their onboarding status. But first, you need to **add a custom claim to the session token** that will allow you to access the user's public metadata in your Middleware.

## The Fix

### ⚠️ CRITICAL: Configure Clerk Dashboard

1. Go to Clerk Dashboard → **Sessions**
2. Under **Customize session token**, add this to Claims editor:
   ```json
   {
     "publicMetadata": "{{user.public_metadata}}"
   }
   ```
3. Click **Save**

### Then Test

1. **Sign out completely** (old tokens won't have the claim)
2. **Sign back in** (get a fresh token with the new claim)
3. Try onboarding flow
4. Check terminal logs - you should see:
   ```
   [Middleware Debug] {
     hasSessionClaims: true,
     hasPublicMetadata: true,  // ← Should be true now
     publicMetadata: { onboardingComplete: true },
     isOnboarded: true
   }
   ```

## Why We Were Confused

1. **Main works** → Because it has the fallback fetch code
2. **Our simplified version breaks** → Because we removed the fallback but didn't configure the session token
3. **Clerk's docs show simple code** → Because they assume you followed Step 1 (configure session token)

## Decision: What Should We Do?

### Option A: Configure Dashboard (Proper Way) ✅
- Configure session token to include `publicMetadata`
- Keep the simple 50-line middleware
- Fast (JWT-based, no extra API calls)
- Follows Clerk's official pattern

### Option B: Keep Fallback Code (Main Branch Way) ❌
- Keep the 110-line middleware with fallback fetches
- Works without Dashboard configuration
- Slow (API call on every middleware request)
- Not following Clerk's recommended pattern

## Recommendation

**Go with Option A**. The Dashboard configuration is a one-time setup. Once done:
- Middleware is fast (JWTs are cheap)
- Code is simple (50 lines vs 110 lines)
- Follows official Clerk patterns
- No API calls on every request
